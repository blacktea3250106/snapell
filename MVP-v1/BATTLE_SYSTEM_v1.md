## v1 修改：心跳簡化

### 原 MVP 方案

```
客戶端每 5 秒寫入 heartbeat → 15s 警告 → 30s 判負
```

### v1 簡化方案

```
不需要獨立心跳機制。
斷線判定改為：
  1. 選卡超時（8 秒）→ 系統隨機選手牌
  2. 連續 3 次選卡超時 → 判負
  3. 超過 60 秒未提交 move（含網路延遲緩衝）→ 判負
```

**省掉：**
- heartbeat subcollection 的讀寫
- 心跳偵測的 Cloud Function 或 scheduled trigger
- 「對手可能斷線」的 UI 狀態

**代價：**
- 無法即時偵測「對手已離開但還沒超時」的狀態
- 可接受：v0→v1 階段是朋友之間打，斷線後直接重開就好

## v1 修改：動畫簡化

### 原 MVP 方案（每回合 ~17 秒，7 段動畫）

```
抽牌(1s) → 場地揭示(2s) → 選卡(8s) → 翻卡(1.5s)
→ 場地效果(1s) → 技能發動(1.5s) → 傷害結算(2s) → 回合結束(1s)
```

### v1 簡化方案（每回合 ~14 秒，3 段）

```
抽牌+場地揭示(2s) → 選卡(8s) → 翻卡+結果(4s)
```

**翻卡+結果（4 秒）合併為一個動畫：**
1. 雙方卡片同時翻開（0.5s）
2. 場地效果 + 技能效果 + 傷害一起顯示（2s）
   - 文字提示：「閃電戰：傷害 ×2」「肉球連擊觸發」
   - HP 條同步下降
3. 回合結束（1.5s）

**本質不變，只是動畫分段合併。** 結算邏輯仍然是完整的 Phase 0.5~4。

## v1 新增：同步機制簡化

### Riverpod StreamProvider

與原 MVP 完全一致：

```dart
final battleProvider = StreamProvider.autoDispose.family<BattleDocument, String>(
  (ref, battleId) {
    return FirebaseFirestore.instance
        .collection('battles')
        .doc(battleId)
        .snapshots()
        .map((snapshot) => BattleDocument.fromFirestore(snapshot));
  },
);
```

### 延遲處理（簡化版）

| 延遲 | v1 處理 |
|------|--------|
| < 2s | 正常 |
| 2~10s | 「等待對手...」文字 |
| > 10s | 「對手似乎斷線了」+ 等待超時自動判負 |

原 MVP 的 5 級延遲分級簡化為 3 級。
